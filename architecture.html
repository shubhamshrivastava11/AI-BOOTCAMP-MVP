Design Assumptions (explicit)

  • Persistence: sessionStorage is a deliberate MVP constraint — no auth or database. State key: dwelliq_onboarding_v1.
  • Affiliate tracking: First-party /api/click route handler owns partner IDs via env vars. SKU is the only thing in the browser URL.
  • Scale: Vercel hobby/pro; hundreds of concurrent users; 200-SKU catalog stays in-bundle. Next migration point: ~2,000 SKUs.

Stateless vs stateful. The recommendation engine is fully stateless — the same OnboardingData object always produces the same RecommendationSet[]. The only stateful layer is the browser: React useState holds the current session's budget totals and category selections; sessionStorage holds the five onboarding answers under the versioned key dwelliq_onboarding_v1. Nothing persists server-side. This is intentional — no database means no auth surface, no GDPR scope, and zero latency on the results page.

Edge vs origin. The homepage and all version pages (/v2–/v5) are statically generated at build time and served instantly from Vercel's CDN edge. The /design and /recommendations pages are client-side rendered — their output depends entirely on browser state, so SSR would add round-trip cost with no benefit. The only origin function is /api/click, which is thin enough to run at the edge.

Security boundary. The most load-bearing line in the architecture is the boundary around the browser. The CATALOG in catalog.ts contains affiliateUrl placeholder values (#affiliate-sofa-001). Real affiliate partner tags live in Vercel environment variables and are only assembled server-side in the /api/click route handler. This prevents partner ID scraping, enables tag rotation without code deploys, and stops open-redirect attacks via SKU allowlist validation.

Section 02

Data Flow

Eleven steps from the first browser request to the affiliate redirect — using the actual file paths, function names, and data shapes from the codebase.

01
Static page load — Vercel CDN edge Edge

User hits dwelliq.app/. Vercel serves the pre-rendered HTML shell from the CDN edge — built at deploy time via Next.js static generation. No SSR work. The Three.js hero canvas is skipped on the server entirely (dynamic(..., {ssr: false})). Google Fonts, the JS bundle, and Unsplash product images load in parallel. Target LCP: <2s on a 4G connection.

Error path

CDN cold miss on first deploy to a new region → Vercel origin renders; ~300–500ms extra. Mitigated by ISR warm-up. Unsplash image 403 → onError handler in RecommendationCard.tsx renders a fallback emoji placeholder.

02
JS hydration — catalog and engine load into memory Client

React hydrates the static HTML. catalog.ts (200 SKUs) and recommendations.ts are imported as ES modules — both ship in the main JS bundle. No lazy-loading for these because every user who reaches /recommendations needs them. Three.js is lazy (ssr:false) and loads only if the hero section is in view. Target bundle: <150 KB gzipped for the main chunk.

03
User starts onboarding — /design Client

User clicks "Design My Room". Next.js router navigates to /design, which renders OnboardingFlow.tsx. The component checks sessionStorage for an existing dwelliq_onboarding_v1 key — if found, it restores progress so a page refresh doesn't lose their answers.

Error path

Safari private browsing disables sessionStorage — the setItem call throws a SecurityError. Without a try/catch wrapper this crashes the entire React tree. Fix: wrap all sessionStorage access in a try/catch with a module-level memory fallback variable.

04
Onboarding inputs collected and persisted Client

The five-step wizard collects: room, budget, style, household, postcode. Each step appends to the sessionStorage object. Validation: budget must be ≥ $500 (below this, the A/B split becomes meaningless). Style must be one of the five StylePreference union literals from types.ts.

// Shape written to sessionStorage at the end of /design
{
  "room":         "living-room",
  "budget":       3000,
  "style":        "warm-mid-century",
  "household":    "couple",
  "postcode":     "10001",
  "primaryUse":   "relaxing",
  "existingPieces": "none",
  "dimensions":   { "length": 14, "width": 11 }
}
05
Budget allocation — CATEGORY_BUDGET_ALLOCATION Client

On "Get Recommendations", the engine applies the hardcoded allocation table to produce per-category budget caps. These ratios are based on interior design industry conventions and are the first parameter the scoring function uses.

// CATEGORY_BUDGET_ALLOCATION (from catalog.ts) — applied to $3,000 budget
sofa:         0.35  → $1,050  cap
area-rug:     0.15  → $450    cap
coffee-table: 0.12  → $360    cap
accent-chair: 0.12  → $360    cap
floor-lamp:   0.08  → $240    cap
// side-table, wall-art, plant, curtains are in catalog but not MVP_CATEGORIES
06
Recommendation engine: filter + scoreProduct() Client

For each of the five MVP categories (sofa, coffee-table, area-rug, floor-lamp, accent-chair), the engine filters the catalog and scores every matching product. The scoring function: style match +3, household suitability +2, product rating (0–5), and a budget-fit score for Option A (penalises products far from 85% of the category cap). Execution: <5ms synchronously in the browser.

function scoreProduct(p, data, targetBudget, mode) {
  let score = 0
  if (p.styles.includes(data.style))           score += 3  // style match
  if (p.householdSuitability.includes(data.household)) score += 2  // household fit
  score += p.rating                                            // 0–5
  if (mode === 'A') {
    const fit = 1 - Math.abs(p.price - targetBudget * 0.85) / targetBudget
    score += fit * 2                                          // budget proximity
  }
  return score
}
Error path

No products pass the style/category filter (rare sparse combinations) → engine removes the style filter and returns top-3 by rating. This guarantees the results page always renders. The loose match is logged as a console.warn in development for catalog gap tracking.

07
Option A / B / C selection logic Client

Three tiers are selected per category with distinct pool filters before scoring is applied.

// Option A — within budget (≤ categoryBudget × 0.92, non-local)
optionAPool = products.filter(p => !p.isLocal && p.price <= cap * 0.92)

// Option B — stretch (between cap and cap × 1.4, non-local, premium tier preferred)
optionBPool = products.filter(p => !p.isLocal
  && p.price > cap && p.price <= cap * 1.4)

// Option C — local first (isLocal sorted by localDistance), fallback to top-rated global
optionCPool = products.filter(p => p.isLocal)
  .sort((a, b) => (a.localDistance ?? 99) - (b.localDistance ?? 99))
08
Results page renders — /recommendations Client

The router navigates to /recommendations, which reads OnboardingData from sessionStorage and calls generateRecommendations() inline. The A/B/C grid renders using RecommendationCard.tsx. The budget tracker in the header initialises to the sum of all Option A prices and updates in real time as the user swaps selections via React state.

09
User swaps options — live budget recalculation Client

Selecting Option B for the sofa updates selectedOptions state: { sofa: 'B', 'area-rug': 'A', ... }. The budget tracker recalculates synchronously. If the new total exceeds the original budget the tracker renders a warning in rust colour — but the selection is not blocked, because flexibility is a legitimate user intent (Option B exists for this reason).

10
Buy click → /api/click route handler Client → Server

User clicks "View at Amazon →". RecommendationCard.tsx calls window.open('/api/click?sku=sofa-001&option=B', '_blank'). The Next.js Route Handler validates the SKU against the CATALOG allowlist, logs the click, builds the affiliate URL from process.env.AFFILIATE_TAG_AMAZON, and issues a 302 redirect. The affiliate partner tag never appears in the browser.

// src/app/api/click/route.ts (Route Handler — runs server-side)
export async function GET(request: Request) {
  const sku    = searchParams.get('sku')
  const option = searchParams.get('option') ?? 'A'
  const product = CATALOG.find(p => p.id === sku)  // allowlist lookup
  if (!product) return new Response('Unknown SKU', { status: 400 })
  console.log(`[click] sku=${sku} option=${option} ts=${Date.now()}`)
  return Response.redirect(buildRetailerUrl(product), 302)
}

// buildRetailerUrl — partner ID in env var, never in client bundle
const tag = process.env.AFFILIATE_TAG_AMAZON
return `https://amazon.com/s?k=${q}&tag=${tag}`
Error path

Unknown SKU → 400, no redirect (prevents open-redirect attack). Missing AFFILIATE_TAG_* env var → redirect fires without the tag (revenue lost but UX intact — better than failing closed for the user). Retailer site down → user lands on retailer's error page; Dwelliq has no retry.

11
Retailer redirect and attribution External

Browser follows the 302 to the retailer product page with Dwelliq's affiliate tag appended. The retailer sets an attribution cookie (Amazon: 24-hour window; others vary). If the user completes a purchase within the window, the affiliate commission is credited. The session in Dwelliq remains intact if the user navigates back — sessionStorage still holds their context and their saved items persist in React state for the tab's lifetime.

Section 03

Design Tradeoffs

Six decisions made during the Dwelliq build where the alternative was genuinely competitive — and why the chosen path fits an affiliate-first MVP with zero backend.

① Client-side recommendation engine vs server API

✓ Chosen — catalog.ts + recommendations.ts in bundle

Both files ship to the browser. generateRecommendations() runs in <5ms with zero network latency. No API to rate-limit, secure, or cold-start. Recommendations are deterministic — same inputs always produce the same A/B/C set, which makes the product easier to test and debug.

✗ Rejected — server-side /api/recommend endpoint

Would unlock personalization, real-time pricing, and a catalog too large for the bundle. Adds 80–250ms round-trip latency, introduces an API surface to secure, and requires a database or vector store for anything beyond simple filtering.

Scale trigger: When the catalog grows past ~2,000 SKUs, the bundle approach breaks. Design for that migration now by keeping all catalog imports behind the single src/lib/catalog.ts module boundary — the server API can slot in without touching any UI components.

② No Framer Motion — pure CSS animations throughout

✓ Chosen — CSS keyframes + Tailwind transition utilities

All animations (hero particle fade, card hover lift, skeleton shimmer, btn-gold glow) are CSS-only. Zero JS animation overhead. No Framer Motion bundle (~30 KB gzipped). Animations don't block the React render cycle. The shimmer skeleton uses a single @keyframes shimmer block in globals.css.

✗ Rejected — Framer Motion

Would enable spring physics, shared layout transitions, and gesture-driven animations (drag-to-sort, swipe cards). These are compelling for a furniture picker — swipe left/right on options is a natural mobile pattern — but add complexity and bundle size that aren't justified at the MVP stage.

UX impact: The absence of spring animations is barely perceptible on desktop. On mobile, swipe gestures on the A/B/C cards would be a genuine UX win — worth revisiting at v2 with Framer Motion's useMotionValue and useTransform rather than the full library.

③ sessionStorage vs user accounts for state persistence

✓ Chosen — sessionStorage (ephemeral, zero-friction)

No sign-up flow. No email capture. No cookie consent for PII. Users get the full product experience in under 2 minutes. The versioned key (dwelliq_onboarding_v1) means schema changes don't corrupt existing sessions — just bump to v2.

✗ Rejected — Auth + saved lists (Clerk + Supabase)

Saves recommendation lists across devices, enables follow-up email ("your sofa is back in stock"), unlocks cohort analytics. But adds 2–3 weeks of implementation and a sign-up friction that kills conversion for the majority of casual visitors at this stage.

Revenue impact: Users who close the tab lose their curated list. For an affiliate product this is an acceptable loss at MVP — the funnel ends at the retailer click-out, not a return visit. Add auth at v2 when retention data shows users want to come back.

④ First-party /api/click redirect vs direct affiliate href

✓ Chosen — /api/click Route Handler with env var tags

Affiliate partner IDs live in process.env.AFFILIATE_TAG_*. The browser only sees the SKU. Click events are logged server-side with sku, option, and timestamp. Partner ID rotation is one env change + redeploy — no catalog update required.

✗ Rejected — Direct affiliate URLs in client bundle / catalog

Simpler — just render <a href={product.affiliateUrl}>. But the partner tag is in the HTML source. Anyone can copy it and use it in their own links, stealing attribution. If the tag is rotated, every deployed page is stale until users hard-refresh.

Security note: An open-redirect is also prevented: the handler validates the sku parameter against the CATALOG allowlist before constructing the redirect URL. An attacker cannot pass an arbitrary URL through /api/click.

⑤ Tailwind v4 CSS-first config vs Tailwind v3 JS config

✓ Chosen — Tailwind v4 with @theme inline in globals.css

All design tokens (--cream, --gold, --ink, etc.) live in the CSS @theme inline block alongside the global styles. No separate tailwind.config.js file. Faster HMR on token changes. Tokens are usable in both Tailwind utilities and raw var() CSS references.

✗ Rejected — Tailwind v3 with tailwind.config.js

Wider ecosystem documentation and tooling support. Some third-party Tailwind plugins haven't migrated to the v4 plugin API, which could block useful additions (typography plugin, forms plugin). More familiar to engineers joining the project from other Next.js codebases.

Onboarding risk: Engineers who know Tailwind v3 deeply will have a learning curve. The config format, some utility names, and the plugin API all changed. Document the v4 differences explicitly in AGENTS.md — it already warns about this pattern.

⑥ Three.js dynamic import (ssr:false) vs CSS gradient hero

✓ Chosen — Three.js with next/dynamic ssr:false

The hero particle animation runs as a WebGL canvas, deferred entirely to the client. dynamic(() => import('./HeroCanvas'), { ssr: false }) means Three.js never runs on the Node.js server — avoiding the window is not defined crash — and doesn't block the initial HTML render.

✗ Rejected — Pure CSS gradient background

Zero JS cost, no SSR risk, works in every browser. The other landing page versions (v3–v5) demonstrate that CSS-only animated backgrounds can be equally striking. But Three.js is part of the product's technical identity and serves as a portfolio differentiator.

Performance note: Three.js adds ~300 KB gzipped. It must stay behind ssr:false and behind a loading placeholder. If Core Web Vitals become a priority, consider switching to a lightweight CSS particle alternative for the v1 landing page and reserving Three.js for the interactive version pages.
Section 04

Technical Insights

Five non-obvious details from the Dwelliq codebase — where the "obvious" implementation would create real bugs, security gaps, or UX problems.

Security

Affiliate partner IDs must never reach the browser — and the SKU must be an allowlist lookup

Two separate vulnerabilities live in the "simple" approach of rendering <a href={product.affiliateUrl}> directly. First, the partner tag (?tag=dwelliq-20) is exposed in the HTML source and can be harvested by scrapers. Second, if you naively use the sku query param to build the redirect URL without validation, an attacker can craft /api/click?sku=https://evil.com and use Dwelliq's server as an open redirector for phishing.

Correct pattern (already implemented): The handler looks up the SKU in the CATALOG allowlist. If not found → 400. The redirect URL is built exclusively from product.baseUrl (known safe) plus the env var partner tag.

// ❌ Wrong — partner tag in source, no allowlist
<a href="https://amazon.com/dp/B09XYZ?tag=dwelliq-20">Buy</a>

// ✓ Correct — only SKU in browser URL
<a onClick={() => window.open(`/api/click?sku=${product.id}&option=${option}`)}>Buy</a>

// Server: allowlist validation before any redirect
const product = CATALOG.find(p => p.id === sku)
if (!product) return new Response('Unknown SKU', { status: 400 })
Browser Rendering

Three.js must be dynamically imported — importing it at module level crashes the Node.js SSR pass

Next.js App Router server-renders pages on Node.js, which has no window, document, or WebGLRenderingContext. Three.js accesses window in some internal paths at import time. Importing it at the module level — even inside a component file marked "use client" — causes the server render to throw ReferenceError: window is not defined and the build fails.

Correct pattern (already implemented): next/dynamic with ssr: false defers the Three.js import to the browser. The loading prop provides an immediate CSS placeholder so there's no layout shift.

// ❌ Wrong — crashes SSR even inside "use client" files
import * as THREE from 'three'

// ✓ Correct — deferred to client, SSR receives the loading placeholder
const HeroCanvas = dynamic(
  () => import('./HeroCanvas'),
  {
    ssr: false,
    loading: () => <div className="h-screen bg-ink" />
  }
)
Data Modeling

sessionStorage access must be wrapped in try/catch — it throws in Safari private browsing and some enterprise policies

In Safari's private browsing mode and in some enterprise browser policies (common in corp laptops), sessionStorage.setItem() throws a SecurityError synchronously. An uncaught exception inside a React event handler or a useEffect propagates up and crashes the React tree — the user sees a blank white screen with no error message.

Correct pattern: Wrap every read and write in try/catch. On write failure, fall back to a module-level variable. On read failure, treat the session as fresh. This is functionally equivalent for the session duration — the only thing lost is recovery after a page refresh, which is unlikely in private browsing anyway.

// ❌ Wrong — throws SecurityError in private browsing
sessionStorage.setItem('dwelliq_onboarding_v1', JSON.stringify(data))

// ✓ Correct — graceful fallback to module-level variable
let memFallback: OnboardingData | null = null

export function saveOnboarding(data: OnboardingData) {
  try {
    sessionStorage.setItem('dwelliq_onboarding_v1', JSON.stringify(data))
  } catch {
    memFallback = data   // works for the full session duration
  }
}
UX Engineering

Budget recalculation must use committed state — not every keystroke — to prevent the results grid from flickering

If the budget input fires generateRecommendations() on every onChange event, the user sees the entire A/B/C grid re-render with every character they type. Even though the engine runs in <5ms, React's reconciler processes a full 5-category × 3-option re-render, causing visible flicker on mid-range devices. A 300ms debounce is the obvious fix but still produces a jarring "settle" animation after typing stops.

Correct pattern: Separate "input value" from "committed budget". The input field updates local state on every keystroke (instant, no re-render of the grid). The engine only runs on blur or explicit submit.

// ❌ Wrong — full grid re-render on every keystroke
<input onChange={e => setBudgetAndRecompute(e.target.value)} />

// ✓ Correct — two states, recompute only on commit
const [inputValue,     setInputValue]     = useState(budget)
const [committedBudget, setCommittedBudget] = useState(budget)

<input
  value={inputValue}
  onChange={e => setInputValue(e.target.value)}
  onBlur={() => setCommittedBudget(Number(inputValue))}
/>
// useEffect watches committedBudget → calls generateRecommendations()
Browser Rendering

Cormorant Garamond is a web font — loading it incorrectly causes FOUT on every page load

Cormorant Garamond is Dwelliq's primary heading font — it defines the product's luxury feel. If it's loaded with the default display=swap and no explicit size-adjust, every heading flashes in the fallback serif (Georgia or Times) for 100–400ms before swapping to Cormorant. On large headings (3–4rem), the layout shift is jarring and the CLS score suffers.

Correct pattern: Use Next.js's built-in next/font/google with display: 'swap' and a variable name. This self-hosts the font via Vercel's CDN (no third-party Google Fonts request), inlines the font-face declaration in the HTML <head>, and allows preconnect to be skipped entirely. Pair with a size-adjust on the fallback stack to minimise layout shift.

// ✓ Correct — self-hosted via next/font, no Google Fonts request at runtime
import { Cormorant_Garamond } from 'next/font/google'

const cormorant = Cormorant_Garamond({
  subsets: ['latin'],
  weight: ['400', '600'],
  style: ['normal', 'italic'],
  display: 'swap',
  variable: '--font-cormorant',
})
// Apply as className on <html> in layout.tsx → available as CSS var everywhere
